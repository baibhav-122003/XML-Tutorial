XML: Your Data's New Best Friend

Why Get Cozy with XML?

XML isn't just another tech acronym – it's the secret sauce behind many web applications you use every day. From sharing data online to powering complex systems, XML is a versatile tool that helps software communicate effectively. Getting a grip on XML is key, whether you're building websites, mobile apps, or working with backend systems.

XML Unpacked: A Beginner's Guide


XML is like a universal translator for data. It lets you store and move information seamlessly between different software and systems, regardless of the underlying hardware.

What is XML, Anyway?

  • XML stands for eXtensible Markup Language – because *you* get to define the markup!
  • Think of it as a cousin of HTML, but focused on data, not display.
  • Its superpowers include storing, transporting, and describing data in a way anyone (or any machine) can understand.
  • Officially approved by the World Wide Web Consortium (W3C).

XML Doesn't *Do* Stuff (Yet)

Here's the thing: XML itself is just a way to organize information. It's like writing a letter – the letter itself doesn't deliver itself; you need a mail carrier for that.

Here's a simple note written in XML:

<< /span>note>
  << /span>to> Alice << /span>/to>
  << /span>from> Bob << /span>/from>
  << /span>message> Meet me at noon! << /span>/message>
<< /span>/note>

See how the tags describe the data? But to actually *use* this note, you need an application to read, send, or display it. That's where the real magic happens.

XML vs. HTML: A Family Feud

XML and HTML are related, but they have different jobs:

  • XML is all about *what* the data *is*. (Meaning)
  • HTML is about how the data *looks*. (Presentation)
  • HTML uses predefined tags (like `

    `); XML lets *you* create tags.

Tag Team: XML and its Custom Tags

Unlike HTML, XML doesn't have a set vocabulary of tags. *You* define the tags that make sense for your data. This makes XML incredibly flexible.

XML: The Shape-Shifter

XML is designed to be adaptable. Even if you add or remove data later, applications built to handle the old format can usually still understand the core information. This is a huge advantage when dealing with evolving data structures.

XML: The Great Simplifier

  • Sharing data between different systems becomes a breeze.
  • Transporting data across platforms? No sweat.
  • Upgrading systems without losing your precious data.
  • Making data accessible to humans, computers, and even talking robots!

XML: Officially Awesome

Since 1998, XML has been a W3C Recommendation - a big thumbs-up from the web's governing body.


XML Power-Ups: What Can You Do?


XML is a web developer's multi-tool. It's often used to separate data from presentation, making your code cleaner and more flexible. It's also the backbone of data exchange in countless applications.

Data and Display: A Clean Break

XML keeps your data separate from how it's displayed. This means you can use the same XML data to generate different presentations – on a webpage, in a mobile app, or even read aloud by a screen reader. This separation of concerns is a core principle of good design.

Tag-Teaming with HTML

XML and HTML work great together. XML handles the data; HTML handles the look and feel. This dynamic duo powers many interactive websites and web applications.

Data Independence

Imagine updating a website every time the data changes – nightmare! XML lets you store data separately, then pull it into your HTML using a few lines of JavaScript. No more constant HTML editing!

XML Family Dynamics

We use terms like *parent*, *child*, and *sibling* to describe the relationships between elements in the XML tree. Just like in a real family, elements can have children, parents, and siblings at the same level in the hierarchy. And each element can contain text (like "The Cat in the Hat") and attributes (like "genre='Children'").

XML Speaks for Itself

XML is designed to be self-explanatory. The tags themselves describe the data, making it easy for both humans and computers to understand the information. The prolog at the top (e.g., ` `) provides additional information about the document, like the XML version and character encoding.


XML Grammar 101: Keep it Tidy!


XML is strict about its grammar rules. These rules ensure that your XML documents are well-formed and easily understood by machines. Think of it like writing a computer program - precision is key.

Every Family Needs a Root

Every XML document must have one root element that acts like the ancestor of all other elements. It's the top of the family tree, and everything else descends from it. For example, in a document about recipes, the root element might be called ``.

Introducing the Prolog

You might see a line at the beginning of an XML document that looks like this: ` `. This is the XML prolog. It's optional, but recommended, and it must appear at the very beginning. It tells the computer which version of XML you're using and how the characters are encoded (UTF-8 is the most common and usually a good choice).

Closing Time: Tags in Pairs

Unlike HTML, which can sometimes be lenient, XML insists on closing every tag. So, if you open a `` tag, you *must* close it with a `` tag. No exceptions! This keeps things organized and unambiguous.

The prolog itself doesn't need a closing tag. It's not technically part of the XML document itself; it's more like an introduction.

Case Sensitivity Alert

XML is case-sensitive, which means `` is different from ``. And if you open a tag with ``, you must close it with ``. Consistency is key!

Nesting Dolls: Proper Nesting

Just like those Russian nesting dolls, XML elements must be properly nested. If you open tag B inside tag A, you must close tag B *before* you close tag A. This ensures a clear, hierarchical structure.

Quote Me on This: Attribute Values

XML attributes, like `id="123"`, must always have their values enclosed in quotes – either single or double. This helps the computer understand where the attribute value begins and ends.

Special Characters: Entity References

Certain characters, like < and &>, have special meanings in XML. To use them literally, you need to use "entity references": `<` for <, `>` for>, and `&` for &. This prevents the computer from misinterpreting these characters as part of the XML structure.

XML Whispers: Comments

You can add comments to your XML code just like in HTML: ``. Comments are ignored by the computer, but they're useful for explaining your code to yourself or others.

White Space Matters

Unlike HTML, XML preserves white space (spaces, tabs, newlines). So, if you have multiple spaces between words, XML will keep them.

Newlines: The LF Standard

XML uses line feed (LF) characters to represent newlines. This is the standard for Unix and macOS. Windows typically uses carriage return and line feed (CRLF), but you should stick to LF in your XML documents.

Well-Formed XML: The Gold Standard

If your XML document follows all these rules, it's considered "well-formed." A well-formed document is a happy document, ready to be used by your applications.

XML Building Blocks: The Elements


XML elements are the fundamental components of an XML document, just like words are the building blocks of sentences. They contain the actual data and provide its structure.

Anatomy of an XML Element

An XML element is everything from its start tag (including the tag itself) to its end tag. Think of it as a container for your data.

<< /span>name> Alice << /span>/name>

Elements can hold various types of content:

  • Text: Like the "Alice" in the example above.
  • Attributes: Metadata about the element, like `id="123"`.
  • Other elements: Elements can be nested within other elements, forming a tree structure.
  • A mix of the above: Elements can contain a combination of text, attributes, and other elements.
<< /span>user id="123">
  << /span>name> Bob << /span>/name>
<< /span>/user>

In this example:

  • `` has text content ("Bob").
  • `` has element content (``).
  • `` has an attribute (`id="123"`).

Empty Elements: Lightweight Champs

Some elements don't have any content, like a line break in HTML (`
`). In XML, you can write empty elements in two ways: `` or ` `. Both achieve the same result. Empty elements can still have attributes.

XML Naming Rules: Choose Wisely

XML elements must follow these naming conventions:

  • Case-sensitive: `` is different from ``.
  • Start with a letter or underscore: `<_name>` is valid, but `<1name>` is not.
  • No "xml" at the beginning (in any case): `` is invalid.
  • Letters, digits, hyphens, underscores, and periods are allowed: `` is valid.
  • No spaces: `` is invalid. Use underscores or camelCase instead.

Naming Best Practices: Keep it Clean

Follow these tips for creating readable and maintainable XML:

  • Descriptive names: Use names that clearly indicate the element's purpose, like ``.
  • Short and sweet: Avoid overly long names like ``.
  • Avoid hyphens, periods, and colons (except for namespaces): These characters can sometimes cause confusion with XML processors.
  • Be consistent: Choose a naming style (e.g., camelCase, snake_case) and stick with it throughout your document.

XML's Extensibility: Future-Proofing Your Data

One of XML's strengths is its extensibility. You can add new elements to your XML documents without breaking existing applications. Imagine you have an application that reads customer data (name, address). Later, you want to add a phone number. With XML, you can add a new `` element without affecting the application's ability to read the existing name and address data.

XML Element Add-ons: Attributes


Attributes provide additional information about XML elements, like metadata. They are always contained within the start tag of an element.

Quote Unquote: Attributed Values

Attribute values *must* be enclosed in quotes – single or double. If the value itself contains double quotes, use single quotes to enclose it (or escape the double quotes with `"`).

Elements vs. Attributes: The Showdown

Should you use an element like `female` or an attribute like `gender="female"`? Both are valid and convey the same information. There are no hard-and-fast rules, but generally, attributes are best for metadata (information *about* the data) while elements are for the actual data itself.

A Matter of Style: Nested Elements

Sometimes it makes sense to use nested elements instead of a long string of attributes. For example, representing a date as ` 2024 10 ` is often clearer and more flexible than `date="2024-10"`.

Attribute Abuse: When Less is More

Avoid overloading elements with too many attributes. If you find yourself adding many attributes, consider using nested elements for better structure and readability. XML is about clear, structured data – don't let a jumble of attributes get in the way.

Attributes for Metadata: The Perfect Match

Attributes are great for storing metadata – information *about* the data, rather than the data itself. For example, an `id` attribute can be used to uniquely identify an element, just like in HTML. This `id` isn't part of the element's core data, but rather helps to identify and refer to it.

XML Name Clash? Namespaces to the Rescue!


Imagine two people named "Bob" in the same room – how do you tell them apart? XML Namespaces solve a similar problem with XML elements. They prevent naming conflicts when combining XML documents from different sources.

Name Collision: When Tags Collide

XML allows you to create custom tags, but what if two different XML documents use the same tag name for different purposes? For example, `

` could represent an HTML table in one document and a furniture table in another. Namespaces prevent these naming collisions.

Prefixes: Giving Tags Nicknames

Namespaces use prefixes to give tags unique identifiers. For example, `` could represent an HTML table, while `` could represent a furniture table. The prefixes ("h" and "f") prevent confusion.

xmlns: Declaring Your Namespace

You declare a namespace using the `xmlns` attribute. For example, `xmlns:h="http://www.w3.org/TR/html4/"` associates the "h" prefix with the HTML namespace. Now, any tag using the "h" prefix (like ``) is clearly identified as belonging to the HTML namespace.

URI: A Unique Identifier

The URI (Uniform Resource Identifier) in the `xmlns` declaration, like "http://www.w3.org/TR/html4/", acts as a unique identifier for the namespace. It's like a family name. The parser doesn't actually *use* the URI to retrieve information, but it helps distinguish between different namespaces.

Default Namespaces: Simplifying Your Code

If you're using a namespace extensively, you can set a default namespace for an element. This saves you from having to use a prefix on every tag within that element. For example, `xmlns="http://www.w3.org/TR/html4/"` sets the default namespace to HTML. Now, any tag without a prefix within that element is automatically part of the HTML namespace.

Namespaces in Action: XSLT Transformation

Namespaces are commonly used in XSLT (Extensible Stylesheet Language Transformations), which is a language for transforming XML documents. The XSLT namespace identifies XSLT elements within a document, allowing you to apply transformations without confusing them with regular XML elements.

Data Fetching Magic: XML Http Request


Modern browsers have a built-in tool called XMLHttpRequest (XHR) that lets you fetch data from a server without refreshing the entire page. It's the engine behind AJAX (Asynchronous JavaScript and XML), which powers many dynamic web applications.

XMLHttpRequest allows you to:

  • Fetch data from the server *after* the initial page load, enabling dynamic content updates.
  • Send data to the server silently in the background, allowing for seamless user interactions.

Making the Request: XHR Code

Here's a typical JavaScript snippet for using XMLHttpRequest:

var xhttp = new XMLHttpRequest();
xhttp.onreadystatechange = function() {
    if (this.readyState == 4 && this.status == 200) {
       // Action to perform when data is ready
       document.getElementById("demo").innerHTML = xhttp.responseText;
    }
};
xhttp.open("GET", "data.xml", true);
xhttp.send();

Breaking Down the XHR Code

  1. Create an XMLHttpRequest object: `new XMLHttpRequest()`
  2. Define what to do when the server responds: `onreadystatechange`
  3. Check if the request is complete and successful: `readyState == 4` and `status == 200`
  4. Update the webpage with the server's response: `responseText`
  5. Open the request (specify method, URL, async): `open("GET", "data.xml", true)`
  6. Send the request: `send()`

XML Decoder: The Parser


Browsers have a built-in XML parser – a tool that reads XML and makes it usable by your JavaScript code. It transforms raw XML text into a structured object that you can easily navigate and manipulate.

XML Parser: Making Sense of XML

The XML DOM (Document Object Model) provides methods to interact with XML data, but first, the XML needs to be parsed – transformed from plain text into a DOM tree. The browser's built-in parser handles this for you.

From Text to Tree: Parsing a String

Here's how you parse an XML string in JavaScript:

const parser = new DOMParser();
const xmlString = "Hello, world!";
const xmlDoc = parser.parseFromString(xmlString, "text/xml");

Now `xmlDoc` holds a DOM object representing the parsed XML, ready for manipulation with DOM methods like `getElementsByTagName()`.

XMLHttpRequest's Built-in Parser

XMLHttpRequest has its own XML parser! The `responseXML` property directly provides the parsed XML as a DOM object. No need for a separate `DOMParser`.

Fetch "books.xml" and use the parsed response:

// ... XMLHttpRequest code ...
const xmlDoc = xhttp.responseXML;
// Now use xmlDoc with DOM methods

XML DOM: Your Data's Control Panel


DOM: The Document Whisperer

The DOM (Document Object Model) is like a universal translator for web documents. It lets your code interact with both HTML and XML in a consistent way, turning them into structured, accessible objects. Think of it as a control panel for your document's content.

"The W3C Document Object Model (DOM) is a platform and language-neutral interface. It allows programs and scripts to dynamically access and update a document's content, structure, and style."

The HTML DOM lets you control HTML; the XML DOM handles XML. They work similarly, representing the document as a tree of nodes that you can manipulate.

Understanding the DOM is essential for web development, whether you're working with HTML, XML, or both.

HTML DOM: Manipulating Web Pages

The HTML DOM lets you change content, add elements, modify styles – virtually anything on a web page.

XML DOM: Taming XML Data

Just like the HTML DOM, the XML DOM lets you access and modify XML data – get element content, change attribute values, add or remove nodes. The XML DOM treats your XML document like a tree, making it easy to navigate and manipulate its structure.

XML Navigation: Using XPath


XPath: The XML GPS

XPath is like a GPS for XML documents. It lets you pinpoint specific elements, attributes, or even text within the complex hierarchy of an XML tree. It's a crucial part of XSLT and other XML technologies.

XPath uses path expressions – think of them as directions – to navigate through an XML document. It has a library of functions and is supported by many programming languages.

XPath and XSLT: A Powerful Duo

XPath is a core component of XSLT (Extensible Stylesheet Language Transformations). XSLT uses XPath to select parts of an XML document for transformation. So, learning XPath unlocks the full potential of XSLT.

XPath by Example

Let's say you have an XML document representing a music library. XPath lets you select things like "all artists whose names start with 'A'," or "the third song in the album 'Dark Side of the Moon'."

Here's a simple example: `/musiclibrary/album[1]/song[2]` selects the second song in the first album. XPath is concise and powerful.

XML Data Whisperer: XQuery


XQuery: Asking Questions of Your XML

XQuery is XML's query language – think of it as the SQL of the XML world. It lets you ask specific questions of your XML data and get back the information you need.

XQuery Example

// Find all books with price > 30
for $x in doc("books.xml")/bookstore/book
where $x/price>30
return $x/title

XQuery is designed specifically for querying XML data. It's supported by most major databases and is a W3C recommendation.

XQuery's Power: Data Extraction

XQuery is all about extracting the data you need from your XML documents. For example, you could use XQuery to find all customers in a certain city from a customer database stored as XML.

XPath and XQuery: Two Sides of the Same Coin

XQuery 1.0 and XPath 2.0 share the same data model and support the same functions and operators. If you have already studied XPath you will have no problems with understanding XQuery.

XQuery - Examples of Use

XQuery can be used to:

  • Extract information to use in a Web Service
  • Generate summary reports
  • Transform XML data to XHTML
  • Search Web documents for relevant information

XQuery is a W3C Recommendation

XQuery is compatible with several W3C standards, such as XML, Namespaces, XSLT, XPath, and XML Schema.

XML, XLink and XPointer


XLink is used to create hyperlinks in XML documents.

  • XLink is used to create hyperlinks within XML documents
  • Any element in an XML document can behave as a link
  • With XLink, the links can be defined outside the linked files
  • XLink is a W3C Recommendation

XLink Browser Support

There is no browser support for XLink in XML documents.

However, all major browsers support XLinks in SVG.

XLink Syntax

In HTML, the <a> element defines a hyperlink. However, this is not how it works in XML. In XML documents, you can use whatever element names you want - therefore it is impossible for browsers to predict what link elements will be called in XML documents.

Below is a simple example of how to use XLink to create links in an XML document:

<?xml version="1.0" encoding="UTF-8"?>

<homepages xmlns:xlink="http://www.w3.org/1999/xlink">
  <homepage xlink:type="simple" xlink:href="https://www.w3schools.com">Visit W3Schools</homepage>
  <homepage xlink:type="simple" xlink:href="http://www.w3.org">Visit W3C</homepage>
</homepages>

To get access to the XLink features we must declare the XLink namespace. The XLink namespace is: "http://www.w3.org/1999/xlink".

The xlink:type and the xlink:href attributes in the <homepage> elements come from the XLink namespace.

The xlink:type="simple" creates a simple "HTML-like" link (means "click here to go there").

The xlink:href attribute specifies the URL to link to.

XLink Example

The following XML document contains XLink features:

<?xml version="1.0" encoding="UTF-8"?>

<bookstore xmlns:xlink="http://www.w3.org/1999/xlink">

<book title="Harry Potter">
  <description
  xlink:type="simple"
  xlink:href="/images/HPotter.gif"
  xlink:show="new">
  As his fifth year at Hogwarts School of Witchcraft and
  Wizardry approaches, 15-year-old Harry Potter is.......
  </description>
</book>

<book title="XQuery Kick Start">
  <description
  xlink:type="simple"
  xlink:href="/images/XQuery.gif"
  xlink:show="new">
  XQuery Kick Start delivers a concise introduction
  to the XQuery standard.......
  </description>
</book>

</bookstore>

Example explained:

  • The XLink namespace is declared at the top of the document (xmlns:xlink="http://www.w3.org/1999/xlink")
  • The xlink:type="simple" creates a simple "HTML-like" link
  • The xlink:href attribute specifies the URL to link to (in this case - an image)
  • The xlink:show="new" specifies that the link should open in a new window

If we set the value of the xlink:show attribute to "embed", the linked resource should be processed inline within the page. When you consider that this could be another XML document you could, for example, build a hierarchy of XML documents.

XPointer

  • XPointer allows links to point to specific parts of an XML document
  • XPointer uses XPath expressions to navigate in the XML document
  • XPointer is a W3C Recommendation

XPointer Browser Support

There is no browser support for XPointer. But XPointer is used in other XML languages.

XPointer Example

In this example, we will use XPointer in conjunction with XLink to point to a specific part of another document.

We will start by looking at the target XML document (the document we are linking to):

<?xml version="1.0" encoding="UTF-8"?>

<dogbreeds>

<dog breed="Rottweiler" id="Rottweiler">
  <picture url="https://dog.com/rottweiler.gif" />
  <history>The Rottweiler's ancestors were probably Roman
  drover dogs.....</history>
  <temperament>Confident, bold, alert and imposing, the Rottweiler
  is a popular choice for its ability to protect....</temperament>
</dog>

<dog breed="FCRetriever" id="FCRetriever">
  <picture url="https://dog.com/fcretriever.gif" />
  <history>One of the earliest uses of retrieving dogs was to
  help fishermen retrieve fish from the water....</history>
  <temperament>The flat-coated retriever is a sweet, exuberant,
  lively dog that loves to play and retrieve....</temperament>
</dog>

</dogbreeds>

Note that the XML document above uses id attributes on each element!

So, instead of linking to the entire document (as with XLink), XPointer allows you to link to specific parts of the document. To link to a specific part of a page, add a number sign (#) and an XPointer expression after the URL in the xlink:href attribute, like this: xlink:href="https://dog.com/dogbreeds.xml#xpointer(id('Rottweiler'))". The expression refers to the element in the target document, with the id value of "Rottweiler".

XPointer also allows a shorthand method for linking to an element with an id. You can use the value of the id directly, like this: xlink:href="https://dog.com/dogbreeds.xml#Rottweiler".

  Anton is my favorite dog. He has won a lot of.....
  </description>
  <fact xlink:type="simple" xlink:href="https://dog.com/dogbreeds.xml#Rottweiler">
  Fact about Rottweiler

XML Errors Will Stop You

Errors in XML documents will stop your XML applications.

The W3C XML specification states that a program should stop processing an XML document if it finds an error. The reason is that XML software should be small, fast, and compatible.

HTML browsers are allowed to display HTML documents with errors (like missing end tags).

With XML, errors are not allowed.

Valid XML Documents

A "well formed" XML document is not the same as a "valid" XML document.

A "valid" XML document must be well formed. In addition, it must conform to a document type definition.

There are two different document type definitions that can be used with XML:

  • DTD - The original Document Type Definition
  • XML Schema - An XML-based alternative to DTD

A document type definition defines the rules and the legal elements and attributes for an XML document.


XML DTD


An XML document with correct syntax is called "Well Formed".

An XML document validated against a DTD is both "Well Formed" and "Valid".

What is a DTD?

DTD stands for Document Type Definition.

A DTD defines the structure and the legal elements and attributes of an XML document.

Valid XML Documents

A "Valid" XML document is "Well Formed", as well as it conforms to the rules of a DTD:

<?xml version="1.0" encoding="UTF-8"?>
<!DOCTYPE note SYSTEM "Note.dtd">
<note>
<to>Tove</to>
<from>Jani</from>
<heading>Reminder</heading>
<body>Don't forget me this weekend!</body>
</note>

The DOCTYPE declaration above contains a reference to a DTD file. The content of the DTD file is shown and explained below.

XML DTD

The purpose of a DTD is to define the structure and the legal elements and attributes of an XML document:

Note.dtd:

<!DOCTYPE note
[
<!ELEMENT note (to,from,heading,body)>
<!ELEMENT to (#PCDATA)>
<!ELEMENT from (#PCDATA)>
<!ELEMENT heading (#PCDATA)>
<!ELEMENT body (#PCDATA)>
]>

The DTD above is interpreted like this:

  • !DOCTYPE note -  Defines that the root element of the document is note
  • !ELEMENT note - Defines that the note element must contain the elements: "to, from, heading, body"
  • !ELEMENT to - Defines the to element to be of type "#PCDATA"
  • !ELEMENT from - Defines the from element to be of type "#PCDATA"
  • !ELEMENT heading  - Defines the heading element to be of type "#PCDATA"
  • !ELEMENT body - Defines the body element to be of type "#PCDATA"

Tip: #PCDATA means parseable character data.

Using DTD for Entity Declaration

A DOCTYPE declaration can also be used to define special characters or strings, used in the document:

Example

<?xml version="1.0" encoding="UTF-8"?>

<!DOCTYPE note [
<!ENTITY nbsp "&#xA0;">
<!ENTITY writer "Writer: Donald Duck.">
<!ENTITY copyright "Copyright: W3Schools.">
]>

<note>
<to>Tove</to>
<from>Jani</from>
<heading>Reminder</heading>
<body>Don't forget me this weekend!</body>
<footer>&writer;&nbsp;&copyright;</footer>
</note>
Try it Yourself »

Tip: An entity has three parts: it starts with an ampersand (&), then comes the entity name, and it ends with a semicolon (;).

When to Use a DTD?

With a DTD, independent groups of people can agree to use a standard DTD for interchanging data.

With a DTD, you can verify that the data you receive from the outside world is valid.

You can also use a DTD to verify your own data.

If you want to study DTD, please read our DTD Tutorial.

When NOT to Use a DTD?

XML does not require a DTD.

When you are experimenting with XML, or when you are working with small XML files, creating DTDs may be a waste of time.

If you develop applications, wait until the specification is stable before you add a DTD. Otherwise, your software might stop working because of validation errors.


XML Schema


An XML Schema describes the structure of an XML document, just like a DTD.

An XML document with correct syntax is called "Well Formed".

An XML document validated against an XML Schema is both "Well Formed" and "Valid".

XML Schema

XML Schema is an XML-based alternative to DTD:

<xs:element name="note">

<xs:complexType>
  <xs:sequence>
    <xs:element name="to" type="xs:string"/>
    <xs:element name="from" type="xs:string"/>
    <xs:element name="heading" type="xs:string"/>
    <xs:element name="body" type="xs:string"/>
  </xs:sequence>
</xs:complexType>

</xs:element>

The Schema above is interpreted like this:

  • <xs:element name="note"> defines the element called "note"
  • <xs:complexType> the "note" element is a complex type
  • <xs:sequence> the complex type is a sequence of elements
  • <xs:element name="to" type="xs:string"> the element "to" is of type string (text)
  • <xs:element name="from" type="xs:string"> the element "from" is of type string
  • <xs:element name="heading" type="xs:string"> the element "heading" is of type string
  • <xs:element name="body" type="xs:string"> the element "body" is of type string

XML Schemas are More Powerful than DTD

  • XML Schemas are written in XML
  • XML Schemas are extensible to additions
  • XML Schemas support data types
  • XML Schemas support namespaces

Why Use an XML Schema?

With XML Schema, your XML files can carry a description of its own format.

With XML Schema, independent groups of people can agree on a standard for interchanging data.

With XML Schema, you can verify data.

XML Schemas Support Data Types

One of the greatest strengths of XML Schemas is the support for data types:

  • It is easier to describe document content
  • It is easier to define restrictions on data
  • It is easier to validate the correctness of data
  • It is easier to convert data between different data types

XML Schemas use XML Syntax

Another great strength about XML Schemas is that they are written in XML:

  • You don't have to learn a new language
  • You can use your XML editor to edit your Schema files
  • You can use your XML parser to parse your Schema files
  • You can manipulate your Schemas with the XML DOM
  • You can transform your Schemas with XSLT

If you want to study XML Schema, please read our XML Schema Tutorial.

XML on the Server


XML files are plain text files just like HTML files.

XML can easily be stored and generated by a standard web server.

Storing XML Files on the Server

XML files can be stored on an Internet server exactly the same way as HTML files.

Start Windows Notepad and write the following lines:

<?xml version="1.0" encoding="UTF-8"?>
<note>
  <from>Jani</from>
  <to>Tove</to>
  <message>Remember me this weekend</message>
</note>

Save the file on your web server with a proper name like "note.xml".

Generating XML with PHP

XML can be generated on a server without any installed XML software.

To generate an XML response from the server using PHP, use following code:

<?php
header("Content-type: text/xml");
echo "<?xml version='1.0' encoding='UTF-8'?>";
echo "<note>";
echo "<from>Jani</from>";
echo "<to>Tove</to>";
echo "<message>Remember me this weekend</message>";
echo "</note>";
?>

Note that the content type of the response header must be set to "text/xml".

See how the PHP file will be returned from the server.

Generating XML From a Database

XML can be generated from a database without any installed XML software.

To generate an XML database response from the server, simply write the following code and save it as an ASP file on the web server:

<%
response.ContentType = "text/xml"
set conn=Server.CreateObject("ADODB.Connection")
conn.provider="Microsoft.Jet.OLEDB.4.0;"
conn.open server.mappath("/datafolder/database.mdb")

sql="select fname,lname from tblGuestBook"
set rs=Conn.Execute(sql)

response.write("<?xml version='1.0' encoding='UTF-8'?>")
response.write("<guestbook>")
while (not rs.EOF)
response.write("<guest>")
response.write("<fname>" & rs("fname") & "</fname>")
response.write("<lname>" & rs("lname") & "</lname>")
response.write("</guest>")
rs.MoveNext()
wend

rs.close()
conn.close()
response.write("</guestbook>")
%>

See the real life database output from the ASP file above.

The example above uses ASP with ADO.

Transforming XML with XSLT on the Server

This ASP transforms an XML file to XHTML on the server:

<%
'Load XML
set xml = Server.CreateObject("Microsoft.XMLDOM")
xml.async = false
xml.load(Server.MapPath("simple.xml"))

'Load XSL
set xsl = Server.CreateObject("Microsoft.XMLDOM")
xsl.async = false
xsl.load(Server.MapPath("simple.xsl"))

'Transform file
Response.Write(xml.transformNode(xsl))
%>

Example explained

  • The first block of code creates an instance of the Microsoft XML parser (XMLDOM), and loads the XML file into memory.
  • The second block of code creates another instance of the parser and loads the XSL file into memory.
  • The last line of code transforms the XML document using the XSL document, and sends the result as XHTML to your browser. Nice!
